What Is Claude — and Why Sapiens Adopted It

1.1 Overview

Claude is Anthropic's enterprise AI assistant. It is designed for professional and technical work: reading and writing documents, reasoning about code and architecture, analyzing data, automating workflows, and supporting complex decision-making — all through natural language.

Sapiens has adopted Claude as its primary enterprise AI platform across all business units. The decision is driven by:

  • Deep reasoning capability suited to insurance domain complexity and multi-constraint architectural problems
  • Anthropic's enterprise data handling commitments — inputs are not used to train models
  • Flexible surfaces: web, desktop, CLI, IDE plugin, browser extension, and desktop automation
  • Programmable context via Projects and CLAUDE.md, allowing team-level customization
  • Native support for long documents, code analysis, and agentic task execution

1.2 What Claude Is Not

⚠ CAUTION
Claude is a powerful reasoning assistant — not an authority. It can be confidently wrong. All Claude output in production, customer-facing, or legal contexts requires human review before use.

Claude does not have access to Sapiens internal systems unless explicitly connected via an MCP integration. It does not know your JIRA tickets, your codebase, or your customer data unless you provide that information in the session.

Connecting External Services — MCP Integrations

4.1 What Is MCP?

MCP (Model Context Protocol) is Anthropic's open standard for connecting Claude to external tools and services. When a service is connected via MCP, Claude can interact with it directly — reading data, executing actions, and incorporating results into its responses — without requiring you to copy-paste information.

Think of MCP connectors as Claude's hands: they allow Claude to reach into external systems on your behalf.
Connected ServiceWhat Claude Can Do
Google CalendarRead events, check availability, create meetings, summarize your schedule
GmailRead emails, draft replies, search threads, summarize conversations
Google DriveRead and search documents, summarize files, find specific content
SlackRead channels, search messages, draft posts, summarize threads
Jira / AtlassianRead tickets, create issues, update status, search projects
AsanaRead and create tasks, update projects, check priorities
SalesforceRead accounts, opportunities, contacts, and activity logs
GitHubRead repositories, create PRs, review diffs, search code
Microsoft 365Read Outlook, Teams, SharePoint documents (via Graph API)

4.2 Connecting a Service in Claude.ai

  1. Open Claude.ai (web or desktop)
  2. Click your profile icon (top right) → 'Settings'
  3. Navigate to 'Integrations' or 'Connectors' in the left menu
  4. Find the service you want to connect and click 'Connect'
  5. You will be redirected to the service's OAuth authorization page
  6. Sign in with your Sapiens account for that service
  7. Click 'Allow' to grant Claude read/write access as specified
  8. Return to Claude — the service is now available in your sessions
💡 TIP
Once connected, reference a service by name: 'Check my Google Calendar for this week and summarize my meetings' or 'Search Gmail for the latest email from the Migration Studio team'.
⚠ CAUTION
MCP connectors use OAuth — Claude never sees your password. Access can be revoked at any time from Settings > Integrations. For Salesforce and production systems, confirm with your team lead before connecting.

Core Concepts — How Claude Works

5.1 The Context Window

The context window is everything Claude can 'see' in a single session: your prompt, the conversation history, any uploaded files or pasted content, the system prompt from your Project, and Claude's own responses. It is measured in tokens.

Claude's context window on the Sapiens Enterprise plan supports up to 200,000 tokens — roughly 150,000 words or about 500 pages of text.

Rough SizeToken CountWhat it Fits
Small< 4,000 tokensA prompt + short document + response
Medium4,000–32,000 tokensA full codebase module + analysis
Large32,000–100,000 tokensMultiple documents + full conversation history
Very Large100,000–200,000 tokensEntire codebases, book-length documents
⚠ CAUTION
Very large contexts are expensive and can degrade response quality. Always trim to the relevant portion. A focused 5,000-token context almost always produces better output than an unfocused 50,000-token one.

5.2 Projects — Persistent Context Across Sessions

A Project is a workspace in Claude.ai with a persistent system prompt — text Claude reads at the start of every conversation inside it. It solves the blank-slate problem: Claude starts every session already knowing your domain, role, preferences, and rules.

The right mental model: Briefing a senior consultant on day one. Everything they need to be immediately useful — without you repeating yourself every conversation.

What belongs in the Project Context

  • Who you are — your role and what's relevant (2-3 sentences, not a CV)
  • The domain — what system, product, or topic this project covers
  • Claude's role — reviewer, challenger, drafting assistant, rubber duck. Define it explicitly
  • Output preferences — format, length, tone
  • Standing rules — things that always apply: never do X, always assume Y
  • What Claude must never do or assume — the most skipped and most valuable part

What does NOT belong

  • Sensitive data, credentials, customer names
  • Information that changes frequently (put that in the conversation)
  • Everything you know about the topic — only what Claude needs to serve you
  • Generic instructions Claude already follows by default ("be helpful", "be accurate")
The one rule that changes everything: The Project Context is a standing instruction. Every sentence should answer: "What does Claude need to know or do consistently, across every conversation in this project, without me asking?" If the answer is "only sometimes" — it belongs in the conversation, not the context.

Recommended structure: one Project per domain

ProjectContext Focus
Specific product / initiativeStack, conventions, constraints, your role on it
PRD & documentation workRequirements standards, templates, writing style
Architecture & strategyDesign principles, architectural constraints, challenge mode on
Personal / generalMinimal — just name and output format preferences

Creating a Project

  1. Click Projects in the left sidebar → + New Project
  2. Name it clearly (e.g., "Migration Studio Dev", "Underwriting PRD")
  3. Click Edit System Prompt → write the context
  4. Optionally upload reference documents (architecture diagrams, style guides)
  5. All conversations inside the project share this context automatically

5.3 CLAUDE.md — Project Context for Claude Code

CLAUDE.md is the Claude Code equivalent of a Project Context. A Markdown file in your project root that Claude Code reads at the start of every CLI session.

Two files load, not one

FileScopePut Here
~/.claude/CLAUDE.mdGlobal — every session, every projectYour personal preferences, coding style, communication defaults, tools you always use
./CLAUDE.mdProject — this repo onlyStack, architecture, conventions, forbidden actions, team context

Local project file takes precedence on conflicts. Both are always active.

What a Strong CLAUDE.md Contains

SectionWhat to Include
Project OverviewOne paragraph: what this codebase does, who uses it, why it exists
Tech StackLanguages, frameworks, databases, APIs, versions
ArchitectureKey components, data flows, service boundaries
ConventionsNaming patterns, file organization, code style, commit format
FORBIDDEN ActionsFiles/dirs never to modify; commands never to run; APIs never to call
Test & BuildExact commands to run tests, build, lint, and deploy
Key DecisionsImportant architectural choices and their rationale (not just what, but why)
Team ContextWho owns which module; escalation paths; review requirements

Three more things worth knowing

  • @filename imports — modularize CLAUDE.md by referencing other files: @docs/architecture.md. Claude loads the referenced file into context.
  • /memory — view loaded context (CLAUDE.md contents and imported files). To update CLAUDE.md within a session, ask Claude to edit the file directly.
  • Git-track your project CLAUDE.md — commit it to the repo. Every team member gets the same Claude context automatically when they clone the project.

Generating CLAUDE.md

cd /path/to/project
claude
/init
💡 TIP
Keep CLAUDE.md under 200 lines. Every line costs tokens on every request. Files over 200 lines risk having instructions ignored entirely. Be precise — not exhaustive.

Additional CLAUDE.md capabilities

  • Subdirectory CLAUDE.md files — place a CLAUDE.md in any subdirectory (e.g. src/api/CLAUDE.md). It loads lazily only when Claude touches files in that directory — keeping global context lean and rules scoped to where they apply.
  • Force adherence on critical rules — prefix any rule that must never be broken with IMPORTANT: or YOU MUST:. Claude follows these with significantly higher reliability than plain instructions.
  • HTML comments are free — <!-- notes for maintainers --> block comments are stripped before injection into Claude's context. Use them for team notes without spending tokens.
  • @filename imports — reference other files: @docs/architecture.md. Claude loads the referenced file into context (max 5 hops). Good for separating team-level and module-level rules.

CLAUDE.md vs. Auto Memory

Claude Code 2.1+ includes auto memory — Claude writes notes to itself across sessions as it learns about your project. These are distinct:

TypeWho writes itWhat it contains
CLAUDE.mdYou and your team (check into git)Architecture, conventions, forbidden actions, build commands — permanent shared rules
Auto memoryClaude (stored in ~/.claude/projects/)Build commands it discovered, debugging insights, preferences learned during sessions

Use /memory to view auto memory. Say "remember X" to save something to auto memory; say "add this to CLAUDE.md" to promote a rule to the permanent shared context.

5.4 Memory — What Claude Remembers

Claude has no persistent memory between conversations by default. Each new conversation starts fresh. There are four ways to give Claude persistent context:

MechanismScopeBest For
Project System PromptAll conversations in the projectTeam-level context: stack, conventions, domain
CLAUDE.mdAll Claude Code sessions in the projectCodebase-level context: architecture, forbidden actions
Claude Memory (Settings)Your personal conversations everywhereYour personal preferences, background, communication style
Uploaded files in conversationSingle conversationOne-off document analysis or reference material
📝 NOTE
Claude's user-level memory (Settings > Memory) stores facts about you that carry across all your conversations — your role, preferred output format, etc. You can view and edit these at any time from Settings.

5.5 Skills — Creating Custom Claude Workflows

A Skill is a Markdown file that packages a reusable workflow into a command Claude can invoke automatically or that you trigger with /skill-name. Think of Skills as custom slash commands with their own instructions, tool permissions, and context.

Mental model: CLAUDE.md tells Claude how to behave in general. A Skill tells Claude how to perform one specific job — with its own instructions, tool access, and even its own isolated context window.

File structure

.claude/
└── skills/
    ├── code-review/
    │   ├── SKILL.md        # Required — instructions + frontmatter
    │   └── checklist.md    # Supporting file (optional)
    └── fix-issue/
        └── SKILL.md

A skill can be a single file (.claude/skills/deploy.md) or a folder with a SKILL.md inside. The folder form lets you include supporting files the skill can reference.

SKILL.md format

A SKILL.md has two parts: YAML frontmatter (configuration) and a Markdown body (the instructions Claude follows when the skill runs).

---
name: fix-issue
description: Fix a GitHub issue. Use when asked to fix any issue number.
argument-hint: "[issue-number]"
allowed-tools: Bash, Read, Edit, Glob, Grep
---

Fix GitHub issue $ARGUMENTS.

1. Run `gh issue view $ARGUMENTS` to read the issue
2. Search the codebase for relevant files
3. Implement the fix — follow conventions in CLAUDE.md
4. Write a test that reproduces the bug, then verify it passes
5. Commit with message: "fix: <issue title> (#$ARGUMENTS)"

Frontmatter fields

FieldRequiredWhat it does
nameYesThe slash command name: /fix-issue. Lowercase, hyphens only.
descriptionYesWhen Claude should invoke this skill automatically. Write it so Claude recognises the right moment — e.g. "Use when asked to fix any issue number."
argument-hintNoShown in the autocomplete tooltip when you type /fix-issue. Example: "[issue-number]"
allowed-toolsNoTools Claude can use without asking permission when running this skill. Example: Bash, Read, Edit
disable-model-invocationNoSet true to prevent Claude from invoking the skill automatically — only you can trigger it. Use for deployments, commits, and anything with side effects.
contextNoSet fork to run the skill in an isolated subagent context (separate context window). The result is summarised back to your main session.
agentNoWhich subagent type to use when context: fork is set. Example: Explore for read-only exploration.
effortNoThinking depth override for this skill: low / medium / high / max
pathsNoGlob patterns — skill only loads when Claude is working with matching files. Keeps context lean.

Variable substitutions in skill body

VariableReplaced with
$ARGUMENTSEverything typed after the skill name: /fix-issue 1234 → $ARGUMENTS = "1234"
$ARGUMENTS[0], $1First argument (0-based index)
${CLAUDE_SESSION_ID}Current session ID
${CLAUDE_SKILL_DIR}Absolute path to the folder containing SKILL.md — use to reference supporting files

Dynamic context injection

Prefix a shell command with !`...` inside the skill body to run it before Claude sees the skill. The output replaces the placeholder at invocation time:

---
name: pr-review
context: fork
---

## Pull request diff
!`gh pr diff`

## Files changed
!`gh pr diff --name-only`

Review this pull request for logic errors, security issues, and test coverage.

This is preprocessing — the commands run once when the skill is invoked, not something Claude decides to execute.

Invocation

  • Manual: type /fix-issue 1234 in your session
  • Automatic: Claude reads the description field and invokes the skill when the situation matches — unless disable-model-invocation: true is set
  • From another skill or agent: reference by name in instructions

Built-in skills (shipped with Claude Code)

SkillWhat it does

/simplify [focus]Reviews recent changes for code quality — spawns 3 parallel review agents and merges their findings

Where to store skills (priority order)

🏢 SAPIENS EXAMPLE
A /db-change skill that: reads the Jira ticket number from $ARGUMENTS, fetches the ticket via !`jira issue view $ARGUMENTS`, then generates a correctly named DDL/DML change file following Sapiens naming conventions — no manual copy-pasting of ticket IDs or timestamps.

5.6 The Planner — Think Before You Act

The Planner is Claude Code's built-in planning workflow. When invoked, Claude proposes a step-by-step plan for completing a task before writing a single line of code or making a single file change.

Why the Planner Matters

Without planning, Claude Code starts implementing immediately — often in a direction that does not align with your intent, requiring expensive correction cycles. The Planner front-loads alignment and prevents wasted work.

Invoking the Planner

Include propose a plan, think through the approach, or plan only — do not code in your prompt:

ultrathink — analyze the FalkorDB ingestion pipeline and propose a plan for
adding streaming support. Do not write any code yet.

Claude will respond with:

  • A summary of what it found (Explore phase output)
  • A numbered implementation plan with rationale
  • Identified risks and open questions
  • A request for your confirmation before proceeding
✅ DO THIS
Always use the Planner for tasks that touch more than 2 files, introduce new architecture patterns, modify shared utilities, or involve any HITL or confidence routing logic.
✖ AVOID THIS
Asking Claude to 'write the code' without a Plan phase for any non-trivial task. This is the single biggest source of rework and wasted tokens in agentic sessions.

Claude Models — Choosing the Right One

6.1 The Model Tiers

Anthropic offers three model tiers in the Claude 4 family. The right model depends on task complexity, not habit.

Haiku 4.5

⚡ Fastest · Lowest Cost
Simple tasks: summaries, classification, short Q&A, data extraction, bulk operations

Sonnet 4.6

⚖ Balanced · Medium Cost
DEFAULT — 85% of work
Code, drafting, analysis, PRDs, architecture discussions, document review

Opus 4.6

🐌 Slowest · Highest Cost
Complex multi-constraint reasoning, high-stakes architectural design, ambiguous tradeoff analysis
⚠ CAUTION
Opus costs significantly more per token than Sonnet. A single Opus conversation with Extended Thinking can cost 10–20x more than an equivalent Haiku conversation. Use Opus only when the reasoning depth materially changes the output quality.

6.2 Model Decision Tree

→ Is the task a lookup, short summary, data extraction, or classification? Haiku 4.5
→ Is it code review, drafting, analysis, debugging, or an architecture discussion? Sonnet 4.6
→ Does it require deep multi-constraint reasoning with significant ambiguity? Opus 4.6
→ Are you stuck in a loop where Sonnet keeps producing shallow output? Escalate to Opus for that sub-problem, then return to Sonnet
→ Is it a bulk processing job on simple documents? Haiku 4.5

7.5 The 4-Phase Workflow — Explore → Plan → Implement → Commit

Anthropic's recommended workflow for any non-trivial Claude Code task. Skipping phases is the primary source of rework and wasted tokens in agentic sessions.

PhaseWhat you doWhat Claude does

2. PlanAsk for a numbered implementation plan. Press Ctrl+G to open and edit it in your text editor.Proposes steps with rationale, flags risks and open questions
3. ImplementSwitch back to normal mode. Confirm the plan.Codes against the plan, runs tests, fixes failures
4. Commit"Commit with a descriptive message and open a PR."Writes commit message, creates PR via gh
💬 EXAMPLE
Phase 1: "Read src/auth/ and understand how sessions and login work. Do not write any code yet."
Phase 2: "ultrathink — propose a step-by-step plan for adding OAuth2 support. Do not code yet."
Phase 3: "Implement the OAuth flow from your plan. Write tests for the callback handler, run them, fix any failures."
Phase 4: "Commit and open a draft PR."
💡 TIP
Skip the Explore and Plan phases only for small, clearly scoped tasks — a one-file change with obvious intent. Planning is most valuable when the change touches multiple files or you are unfamiliar with the code.

Naming and resuming sessions

Course-correcting

SituationWhat to do
Claude is going in the wrong direction mid-taskPress Esc — stops Claude, preserves context. Redirect with a follow-up prompt.
Claude made a bad change and you want to undo it/rewind — restores the previous conversation + code checkpoint. Claude auto-checkpoints before every change.
Claude has made the same mistake twice after correction/clear — reset context entirely and write a better prompt that incorporates what you learned.
You want a quick side answer without polluting context/btw — answer appears in a dismissible overlay; never enters conversation history.
⚠ RULE OF THUMB
If you have corrected Claude on the same issue more than twice, the context is polluted with failed attempts. Run /clear and start fresh — a better initial prompt will get you there faster than continuing to correct.

Parallel work with Git Worktrees

Run multiple Claude sessions simultaneously without file conflicts — each session gets its own isolated copy of the repository:

claude --worktree feature-auth    # Isolated copy for auth feature
claude --worktree bugfix-123      # Separate isolated copy for the bug fix

Worktrees branch from the default remote branch. If the session makes no changes, the worktree is auto-deleted. Add .claude/worktrees/ to your .gitignore.

7.7 Hooks — Deterministic Lifecycle Actions

Hooks run shell commands automatically at specific points in Claude's lifecycle. Unlike CLAUDE.md instructions (advisory), hooks are deterministic — they guarantee the action happens every time, regardless of what Claude decides.

Key distinction: CLAUDE.md tells Claude what to do. Hooks make it happen unconditionally — even if Claude forgets or decides otherwise.

Common use cases

Hook EventWhen it firesCan block?Example use
PreToolUseBefore any tool executesYesBlock destructive commands (rm -rf, DROP TABLE), validate inputs
PostToolUseAfter a tool succeedsNoRun linter/formatter after every file edit
StopClaude finishes respondingYesQuality gates — reject response if tests fail
NotificationClaude sends a notificationNoDesktop notification when long task completes
SessionStartSession beginsNoLoad env vars, inject context

Configuration

Hooks are defined in .claude/settings.json (project-level, commit to git) or ~/.claude/settings.json (personal, all projects):

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npm run lint --fix", "timeout": 30 }
        ]
      }
    ]
  }
}
💡 TIP
Ask Claude to write hooks for you: "Write a hook that runs eslint after every file edit" or "Write a hook that blocks any write to the migrations folder."

Exit codes for command hooks

  • Exit 0 — Success; execution continues normally
  • Exit 2 — Blocking error; stderr is fed back to Claude as context; Claude adjusts and retries
  • Other — Non-blocking warning; stderr shown in verbose mode; execution continues

Token Economics — Using the AI Budget Wisely

8.1 What Are Tokens?

Tokens are the unit of measurement for AI processing. Roughly 1 token = 0.75 English words. Every word you send to Claude (input) and every word Claude responds with (output) costs tokens. Sapiens pays for these tokens at the organizational level.

ItemApproximate Token Count
1 word~1.3 tokens
1 page of text~500 tokens
1,000 lines of code~2,000–3,000 tokens
Average chat conversation~2,000–8,000 tokens
Large PRD document (20 pages)~10,000 tokens
ultrathink reasoning budget~32,000 tokens (per request)
Full context window (200K)~150,000 words / ~500 pages

8.4 The 5 Golden Rules

1
Right model for the task. Do not use Opus for email drafts. Do not use Haiku for architecture decisions.
2
Trim your context. Paste only the relevant section of a document — not the whole thing.
3

4

5
Invest in the prompt. A specific, well-structured prompt avoids 2-3 correction cycles. Write it once; write it well.

Prompting Fundamentals

9.1 The Anatomy of a Good Prompt

Every effective Claude prompt contains some or all of these elements. You do not need all of them every time — but the more that are relevant, the better the output.

ElementPurposeExample
RoleTell Claude who it should be'You are a senior insurance domain architect'
ContextTell Claude what situation it is in'We are migrating from batch to streaming ingestion in Migration Studio'
TaskState clearly what you want'Identify the failure points in the current design'
ConstraintsDefine what is off-limits or required'Do not suggest changes to the HITL gateway. Keep the response under 500 words.'
FormatSpecify how you want the output'Respond as: (1) one-paragraph summary, (2) bullet list of risks, (3) one recommendation'
ExamplesShow Claude what good output looks likePaste a sample output and say 'Match this format'

9.2 Common Prompting Mistakes

MistakeWhat HappensFix
Too vague: 'Review this'Claude reviews everything superficiallySpecify: what to review, what to look for, what to ignore
No output format specifiedClaude writes a wall of proseAdd: 'Return as a bullet list' or 'Use a table'
Pasting the entire documentClaude spreads analysis thin across all contentPaste only the relevant section + a sentence explaining the rest
Asking too many things at onceClaude addresses all of them shallowlyOne task per prompt; chain prompts for multi-step work
Not specifying the audienceClaude pitches at the wrong levelAdd: 'For a non-technical executive' or 'For a senior developer'
Accepting the first output without iterationMiss the best answerAlways ask: 'Is there a better approach?' or 'What did you not consider?'

9.3 Iteration Patterns

The most productive Claude sessions are iterative. These patterns consistently improve output quality:

  • Challenge the answer: "What are the weaknesses in this approach?" — Claude will critique its own output.
  • Ask for alternatives: "Give me 3 different approaches to this problem and compare their tradeoffs."
  • Narrow the scope: "Focus only on the confidence routing logic — ignore everything else."
  • Elevate or simplify: "Rewrite this for a non-technical stakeholder" or "Add more technical depth."
  • Request a different format: "Convert this to a table" or "Summarize this in 3 sentences."
  • Verify reasoning: "Walk me through your reasoning step by step" — especially useful after ultrathink.

9.4 Give Claude a Way to Verify Its Work

This is the single highest-leverage thing you can do in a Claude Code session. Claude performs dramatically better when it can check whether its output actually works — without verification, you become the only feedback loop.

Type of workVerification to provide
Code changesTell Claude to run the test suite: "Write a failing test that reproduces the bug, then fix it and confirm the test passes."
Build / compilation"After making changes, run the build and fix any errors. Do not suppress errors — fix the root cause."
UI changesPaste a screenshot and ask Claude to compare against the target; or use the Claude in Chrome extension
Bug fixPaste the exact error message: "Fix this error and verify the build succeeds."
Refactor"After refactoring, run all tests and confirm nothing regressed."
✅ DO THIS
When asking Claude to fix a bug: paste the exact error message, tell it to address the root cause (not suppress the error), and ask it to verify the fix by running the build or tests.

9.5 Interview Pattern — For Large Features

For complex features, asking Claude to interview you before coding produces far better output than jumping straight to implementation. The interview surfaces edge cases, tradeoffs, and requirements you have not yet articulated.

I want to build [brief description]. Interview me in detail before writing any code.
Ask about technical implementation, UI/UX, edge cases, concerns, and tradeoffs.
Keep asking until you have everything you need, then write a complete spec to SPEC.md.

Then start a fresh session to implement from the spec. Clean context + written spec consistently produces better output than a long, exploratory session.

9.6 Writer / Reviewer Pattern

For critical code, use two separate Claude Code sessions — one to implement, one to review. The reviewer session has no context pollution from the implementation decisions and catches more issues.

  1. Session A (Writer): "Implement a rate limiter for the API gateway." → Claude writes the code.
  2. Session B (Reviewer): "Review @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with existing patterns. Do not suggest rewrites — identify specific problems."
  3. Session A (Fix): Paste the review feedback → "Address these issues from the review."
💡 TIP
The reviewer session works best with /effort high or ultrathink — it's doing analysis, not generation, and benefits from deeper reasoning.

Security, Data Policy & Acceptable Use

10.1 Data Handling

Sapiens operates Claude under an Enterprise Data Processing Agreement with Anthropic. Key commitments:

  • Inputs are NOT used to train Anthropic's models
  • Data is processed in Anthropic's cloud infrastructure with SOC 2 Type II and ISO 27001 controls
  • No data is stored beyond the session unless you explicitly use Projects or Memory features
  • Usage logs are retained for billing and compliance — accessible to Sapiens IT admins

10.2 What You Must NEVER Paste Into Claude

⚠ CAUTION — NEVER PASTE
The following data types must never be entered into any Claude surface — web, desktop, CLI, Cowork, or Chrome extension:
  • Customer PII: policyholder names, policy numbers, claim IDs, medical data, financial account details
  • Production credentials: API keys, passwords, tokens, connection strings, certificates
  • Production database contents or exports (even anonymized samples require IT approval)
  • Unreleased M&A, financial, or legal documents (without explicit approval from Legal)
  • Third-party confidential data covered by NDA

10.4 Acceptable Use Policy

  • Claude is for Sapiens professional work tasks only. Personal use on company accounts is not permitted.
  • Do not use Claude to draft external communications about unreleased products or pricing without review.
  • Do not share Claude-generated code in production without a human code review.
  • Do not rely on Claude for legal, compliance, or financial advice — always escalate to the appropriate function.
  • Do not attempt to bypass Claude's built-in safety behaviors using prompt engineering techniques.
  • Claude output is AI-generated. All outputs used in customer-facing, legal, or production contexts require human review and approval.
📝 NOTE
